# 調査結果

## 1. Minecraft 音符ブロックの仕様(バニラ)

### 基本動作

- 音符ブロックは**右クリックで音程が1段上がる**。0〜24 クリックの **25 音(2オクターブ、半音階)**
- 24 の次は 0 に戻る(周回)
- **上のブロックが空気でないと鳴らない**(設計書の注意書きに含める)
- レッドストーン信号の立ち上がり(OFF→ON)で発音。持続音は不可(サステインなし)。タイ・長い音符は onset のみ発音する

### 音色(直下のブロックで決まる)

各音色は 0 クリック = 基準音、24 クリック = 基準音+2オクターブ。

| 音色 | 下に置くブロック | 音域 | MIDI 番号 | 用途 |
|---|---|---|---|---|
| **bass** | 木材系(板材など) | F#1〜F#3 | 30〜54 | 低音(左手) |
| didgeridoo | パンプキン | F#1〜F#3 | 30〜54 | 低音の代替音色 |
| guitar | 羊毛 | F#2〜F#4 | 42〜66 | 中低音 |
| **harp** | 土・石以外の多く(デフォルト) | F#3〜F#5 | 54〜78 | 主旋律(ピアノに最も近い) |
| iron_xylophone(鉄琴) | 鉄ブロック | F#3〜F#5 | 54〜78 | harp と同音域の別音色 |
| pling | グロウストーン | F#3〜F#5 | 54〜78 | harp と同音域の別音色 |
| bit | エメラルドブロック | F#3〜F#5 | 54〜78 | 〃 |
| banjo | 干草の俵 | F#3〜F#5 | 54〜78 | 〃 |
| cow_bell | ソウルサンド | F#4〜F#6 | 66〜90 | 中高音 |
| flute | 粘土 | F#4〜F#6 | 66〜90 | 中高音 |
| **bell** | 金ブロック | F#5〜F#7 | 78〜102 | 高音 |
| chime | 氷塊(パックドアイス) | F#5〜F#7 | 78〜102 | 高音の代替 |
| xylophone(木琴) | 骨ブロック | F#5〜F#7 | 78〜102 | 〃 |
| basedrum | 石系 | (打楽器) | — | ドラム(将来拡張) |
| snare | 砂・砂利 | (打楽器) | — | 〃 |
| hat | ガラス | (打楽器) | — | 〃 |

**ピアノ音域(A0=MIDI 21 〜 C8=MIDI 108)は harp の2オクターブを超える**ため、
bass / harp / bell の3音色への振り分け(MIDI 30〜102 をカバー)+範囲外のオクターブシフトで対応する。

### クリック回数の計算

```
clicks = MIDIノート番号 − 音色の基準音のMIDI番号
例: C4 (MIDI 60) を harp (基準 F#3 = MIDI 54) で鳴らす → 60 − 54 = 6 クリック
```

## 2. レッドストーンのタイミング仕様

- **レッドストーンティック(RT)= 0.1 秒**(ゲームティック 2 つ分)
- **リピーターの遅延は 1〜4 RT** に設定可能(右クリックで目盛が変わる。初期値 1)
- 長い遅延はリピーターを直列に並べて**加算**する

### 遅延のリピーター分解

```
遅延 d RT → 4目盛リピーター × (d ÷ 4 の商) + 余りがあれば余り目盛のリピーター1個
例: 7 RT → リピーター2個(4目盛 + 3目盛)
    12 RT → リピーター3個(4目盛 × 3)
```

## 3. タイミング量子化の数理

### 課題

楽譜上の音符の発音タイミングを 0.1 秒(RT)グリッドに乗せる必要がある。
テンポによっては 16 分音符が整数 RT にならない(例: 120 BPM の 16 分音符 = 0.125 秒 = 1.25 RT)。

### 採用方式: 拍グリッド方式(既定)

秒を丸めるのではなく、**music21 の offset(quarterLength = 4分音符単位)× `ticks_per_quarter`(tpq)を整数丸め**する。

```
実効BPM = 600 / tpq
  tpq 3 → 200 BPM
  tpq 4 → 150 BPM
  tpq 5 → 120 BPM
  tpq 6 → 100 BPM
  tpq 8 →  75 BPM
```

- 丸め誤差が蓄積・ドリフトしない(楽譜上の拍位置に対して常に一定比率)
- 16 分音符(0.25 quarterLength)は tpq が 4 の倍数なら誤差 0 で整数 RT に乗る
- 原曲 BPM と一致しない場合は「実効 BPM 150(原曲比 1.5 倍速)」のように UI に明示し、ユーザーが tpq を選ぶ

### 補助方式: 秒グリッド方式

MIDI 内にテンポ変化が多い曲向け。music21 の secondsMap × tempo_scale を 0.1 秒に最近傍丸め。
誤差統計(最大/平均 ms、移動したノート数)をレポートする。

### 同時発音の扱い

- 同一 tick に落ちたノートは同一ステップにまとめる(和音・左手右手の同時発音)
- 同一 tick・同一音色・同一クリック数の重複はデデュープし、警告として報告

## 4. OMR(光学楽譜認識)の候補比較

| 候補 | 実装 | 特徴 | 判定 |
|---|---|---|---|
| **Audiveris** | Java | 成熟した OSS。CLI バッチモード(`Audiveris -batch -export`)で PDF/画像 → MusicXML(.mxl)出力。JRE があれば動くので Docker コンテナ化が容易 | **採用**。compose の別サービスとして分離 |
| oemer | Python | 深層学習ベース。pip で入るが TensorFlow/ONNX など依存が非常に重く、イメージサイズ・メモリ消費が大きい | 見送り(README に代替として記載) |

- OMR の精度は楽譜の品質に大きく依存するため「実験的機能」として提供し、UI に免責を明示する
- 変換後は MusicXML として既存パイプラインにそのまま合流できる(パイプラインの変更不要)

## 5. プレビュー再生(Web Audio)

- Minecraft の実サウンドは著作物のため同梱しない。**Web Audio API による合成音**で代替:
  - harp / pling: 減衰の速い三角波 + ローパスフィルタ
  - bass: 矩形波の低域
  - bell / chime: 倍音を重ねたサイン波 + 長めの減衰
- 再生スケジューリングは lookahead 方式(AudioContext の currentTime に対して先読みしてスケジュール)
- `tick × 0.1 秒 ÷ 再生速度` で発音時刻を決定 → **量子化後のタイミングそのもの**を確認できる
